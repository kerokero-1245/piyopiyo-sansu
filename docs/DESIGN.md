# ぴよぴよさんすう— 設計ドキュメント

> 4歳児のための、超やさしい「たしざん・ひきざん」知育アプリ。
> 数字の記号操作ではなく、**キャラクター（モノ）が増えたり減ったりするのを見て、指でさして数える**グラフィカルなさんすう。
> ひよこが増える → 「ぜんぶで いくつ?」／りんごが食べられる → 「のこりは いくつ?」。
> 正解すると1つずつハイライトしながら「1…2…3!」と数え上げて祝福する。

最終更新: 2026-07-12

同じ作者の第1作 [おつかいめいろ](https://github.com/kerokero-1245/otsukai-meiro)（4歳向け・Web公開済み）と同系統。UX原則・配布方針・技術スタック・部品（大ボタン／紙吹雪／効果音合成／親ゲート）を流用する。

## 0. 配布方針（Webファースト・確定）

**決定: まず Web で配る。**（おつかいめいろと同方針）Expo の web ビルド（静的サイト）を書き出し、その URL を姪っ子の端末（を持つ家族）に渡す。ブラウザで開き、**ホーム画面に追加**すれば、アドレスバーの無い全画面で“アプリのように”遊べる（iOS / Android とも対応）。ネイティブアプリ（App Store / Expo Go）は**同じコードから後日**。

- **なぜ Web ファーストか** … 審査・開発者登録なしで今すぐ実機で試せる／端末を選ばない（URL一つ）／同一コードから将来ネイティブ化も地続き。
- **ホスティング: GitHub Pages**。`npx expo export --platform web` で `dist/` を書き出し、GitHub Actions で自動デプロイ。サブパス（`/piyopiyo-sansu/`）配信のため、ビルド時のみ `EXPO_BASE_URL` を渡してベースパスを付ける（ローカル開発には影響させない）。
- **完全オフライン**: バックエンド・アカウント・通信・外部送信はゼロ。おとなモードの設定（数の範囲）だけ端末内（web は `localStorage` 相当）に保存する。消えても遊びは壊れない設計。

## 1. コンセプト

**数字の記号操作（2 + 1 = 3 を暗記・計算する）を教えない。** 4歳の発達段階で本当に育てたいのは、**数唱**（1,2,3…と唱える）と**1対1対応**（モノ1つに数1つを当てる）。だから本アプリは徹頭徹尾「モノを見て、指でさして、数える」体験にする。

- **たしざん = ふえる** … ひよこが2わ。そこへ1わ **歩いてくる**（増えるアニメ）。「ぜんぶで いくつ?」
- **ひきざん = へる** … りんごが4こ。2こ **食べられて消える**（減るアニメ）。「のこりは いくつ?」
- **答えは数えて出す** … 選択肢は「**●●● ＋ 数字**」を併記した巨大ボタン。数字だけでなく**同じ数の丸（ドット）**を必ず添え、数字が読めなくても“数”で選べるようにする。
- **正解 = 数え上げ演出** … 1つずつハイライトしながら「1…2…3!」と数える。ここが学びの核（数唱＋1対1対応の反復）。最後に祝福。
- **まちがい = 罰なし** … 選んだボタンが**ぷるぷる震える**だけ。そのあと**正しい数え上げをゆっくり見せて**再挑戦。タイマー・スコア・ゲームオーバーは一切なし。

### この子に寄せる差別化（将来）

おつかいめいろと同じ思想で、汎用ドリルには作れない要素を後で足す（§11）。掛け声「せいかい!」「すごいね!」を**おじの録音声**に、キャラを姪っ子の好きなモチーフに。MVPは既定の絵文字＋合成音で成立させる。

### 立ち位置（2つの前提）

| 前提 | 中身 |
|---|---|
| **プライバシー: 完全オフライン** | 単一端末で完結。設定は端末内のみ。**外部送信・アカウント・通信はゼロ**。 |
| **著作権** | 既存キャラIP（アンパンマン等）は不可。素材は**絵文字／自作／AI生成／フリー素材／おじの録音**のみ。MVPは OS の絵文字をプレースホルダに使う。 |

## 3. 問題の種類と数の範囲（本アプリの背骨）

### 二種類の問題

| | たしざん（ふえる） | ひきざん（へる） |
|---|---|---|
| 見せ方 | モノが `a` こ → `b` こ 歩いてくる | モノが `a` こ → `b` こ 食べられて消える |
| 問い | 「ぜんぶで なん◯?」 | 「のこりは なん◯?」 |
| 式（内部） | `a + b = answer` | `a − b = answer` |
| 答え | 数えた総数 | 数えた残り |

1セット5問。たしざん・ひきざんを**混ぜて**出題する（種類を意識させず「数える」だけに集中させる）。

### 数の範囲（4歳の適正レンジ）

- **こどもモード（既定）: 合計5まで。** たしざんは `a≥1, b≥1, a+b≤5`（答え2〜5）。ひきざんは `a≤5, b≥1, answer≥1`（答え1〜4）。→ 答えは常に **1〜5**。
- **おとなモード: 合計10まで**に拡張（同じロジックで上限だけ引き上げる）。数唱・1対1対応が安定してきたら親が切り替える。
- 上限は設定値 `maxSum`（既定5、おとなで10）1つで制御。ロジックは共通。

## 4. 4歳UX原則（全画面の共通ルール）

- **文字に頼らない** … 問いは増減アニメ・数え上げ・アイコンで伝わる構成。ひらがな短文（ふりがなレベル）は“読める子には嬉しい”装飾程度。
- **巨大タップ領域** … 選択肢ボタンは指1本で確実に押せる大きさ。間隔をあけて押し間違いを防ぐ。
- **単発タップ中心** … ドラッグ・長押しはこども操作に使わない（親ゲートを除く）。
- **誤操作に寛容 / 罰なし** … 不正解でもボタンがぷるぷる震えるだけ。タイマー・ライフ・減点・ゲームオーバーなし。まちがい＝「正しい数え方をもう一度見る」入口。
- **数え上げをていねいに** … 正解時・不正解後の再提示とも、1つずつゆっくりハイライトして数える。急がせない。
- **短いセッション** … 1セット5問・数十秒〜。いつでも中断・再開。連続プレイを煽らない。
- **過度な射幸性なし** … ガチャ・コイン・ランキング・広告なし。ご褒美は「数え上げ＋ちいさな紙吹雪＋やさしい音」に限定。
- **感覚にやさしく** … 強い点滅・大音量・急な動きを避ける。音量は控えめを既定に。

## 5. MVP スコープ

### コアループ（これ1つを作り切る）

```
[もんだい] たしざん or ひきざん をランダム出題
  → モノが増える／減えるアニメを見せる
  → 「ぜんぶで/のこりは いくつ?」＋ 3つの巨大選択肢（●●●＋数字）
  → タップ
     ・正解 → 1つずつハイライトして「1…2…3!」→ ちいさな紙吹雪＋やさしい音 → つぎの問題へ
     ・不正解 → 押したボタンがぷるぷる → 正しい数え上げをゆっくり見せる → もう一度えらぶ
  → 5問おわったら [できたね!]（紙吹雪＋「もういちど」/「おうちへ」）
```

### やること（MVPに入れる）

- タイトル画面（あそぶ／親ゲート）
- もんだい画面（増減アニメ＋数え上げ＋3択）— たしざん・ひきざんを混ぜて出題
- 問題のランダム生成（**シード無し**、合計≤5、答え1〜5、選択肢は正解＋近い誤答2つ）
- 正解時／不正解後の数え上げ演出
- 5問1セット → できたね!画面
- おとなモード（親ゲートの奥）: **数の範囲 5まで／10まで の切替**（端末内保存）
- 効果音（web は Web Audio 合成、native は無音スタブ）

### やらないこと（MVP対象外 = 次段階）

- おじの声の録音・差し替え（フェーズ2）
- キャラの差し替え・パーソナライズ（フェーズ2）
- 「◯といくつで10」等の高度な数遊び／繰り上がり（将来）
- 進捗の永続化・レポート（親向け）… 当面なし
- ネットワーク・バックエンド・アカウント全般

## 7. 問題生成ロジック & データモデル

### 問題オブジェクト

```ts
type Op = 'add' | 'sub';
interface Problem {
  op: Op;          // 'add' | 'sub'
  a: number;       // 最初の数（≥1）
  b: number;       // 増える/減る数（≥1）
  answer: number;  // add: a+b / sub: a−b（常に 1〜maxSum）
  char: CharDef;   // 使うキャラ（絵文字＋数え方の語尾）
  choices: number[]; // 選択肢（answer を含む3つ・シャッフル済み）
}
interface CharDef { emoji: string; ask: string; } // ask 例: 'なんわ' 'なんびき' 'なんこ'
```

### 生成手順（`src/game/problems.ts`、シード無しランダム）

1. `op` を 50/50 で選ぶ。
2. **add**: `a`,`b` を `a≥1, b≥1, a+b≤maxSum` からランダム → `answer=a+b`。
   **sub**: `a` を `2..maxSum`、`b` を `1..a-1` → `answer=a−b`（`answer≥1`）。
3. `char` をキャラ表からランダム。
4. 選択肢: `answer` に加え、`1..maxSum` から `answer` に近い誤答を2つ選び、重複を避けてシャッフル（計3つ）。
5. これを5問ぶん作って1セット（`generateSet(maxSum, 5)`）。前問と同じにならない程度の軽い分散はするが、**シード固定はしない**（毎回ちがう）。

**座標も盤面も持たない**（おつかいめいろと違い、モノの並びは描画側が数から自動レイアウト）。ステージは「present な数」を数え上げの対象にする（add=総数、sub=残り）。

### 設定（`src/settings.ts`）

```ts
// key: "sansu.maxSum"  値: 5 | 10（既定 5）
```

web は `localStorage`、native は将来 AsyncStorage。取得失敗時は既定5にフォールバック。**外部送信なし**。

## 8. 技術構成

おつかいめいろと**同一スタック**（Expo SDK 57 / React Native / TypeScript）。バックエンドなし。

| 役割 | 技術 | 用途 |
|---|---|---|
| アプリ基盤 | **Expo SDK 57 / React Native / TypeScript** | web / iOS / Android 同一コード |
| 画面遷移 | 自作の最小ステートルーター（`App.tsx`） | 文字ヘッダの無い全画面遷移（title→play→otona） |
| アニメーション | **React Native 標準 `Animated`** | モノの登場（歩いてくる）・退場（食べられる）・数え上げのポップ・紙吹雪 |
| 効果音 | **Web Audio API 合成**（web）／無音スタブ（native） | タップ・登場/消滅・数える音・正解ファンファーレ・不正解のやさしい音 |
| 設定保存 | `localStorage`（web） | `maxSum` の永続化 |

> **実装前に必ず** Expo SDK 57 公式ドキュメント（`https://docs.expo.dev/versions/v57.0.0/`）で導入手順・API を確認する（記憶で書かない）。アニメは reanimated を足さず**標準 Animated** に統一（依存を増やさない）。

### Web対応の注意（§0 のWebファースト前提）

- **音声の自動再生制限** … ページ読み込み直後に勝手に音は鳴らせない。SE・数える音は必ず**タップ起点**（ボタン押下／その後の演出）で鳴らす。BGM自動ループは避ける。
- **レイアウトが可視領域に必ず収まること（重要・おつかいめいろの教訓）** … もんだい画面は「上段（進捗）＋ステージ＋問い＋選択肢帯」で構成する。**ヘッダー・問い・選択肢帯の高さを差し引いた残り**にステージ（モノの描画）が必ず収まるようサイズ計算する。おつかいめいろでは固定サイズの盤面が背の低いウィンドウでヘッダーに覆いかぶさる不具合（コミット5bed703で修正）が出た。本アプリはステージを `flex:1` で確保し、**実測（onLayout）した残り領域に合わせてモノの大きさを算出**して、絶対にあふれさせない。完了前に Playwright で複数ビューポート（1280×800 / 1030×1368 / 515×684 / 390×844）の目視確認を必須にする。
- **端末内保存の永続性** … `localStorage` はプライベートブラウズ・データ削除で消え得る。設定が消えても既定5で成立するので許容。
- **画面向き** … 縦持ち前提（中央寄せ・最大幅制限）で横画面でも崩れないように。

### 開発環境メモ

- Node は **v20（.nvmrc: 20.20.0）**。CI（Actions）も Node 20。
- リポジトリ名は `piyopiyo-sansu`。
